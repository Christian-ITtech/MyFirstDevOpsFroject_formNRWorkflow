## PARTIE 2 : ANALYSE & FONCTIONNEMENT DU BACKEND WEB## Le concept fondamental du Backend
Votre premier script Python utilisait la fonction input() pour lire le clavier dans le terminal et print() pour y afficher un texte.
Sur un site web, la logique reste la même, mais les tuyaux changent :

   1. La Requête (Frontend $\rightarrow$ Backend) : Le navigateur de l'utilisateur envoie les données (le mot de passe ou le formulaire) via une requête réseau. input() est remplacé par request.form (ou request.get_json()).
   2. La Réponse (Backend $\rightarrow$ Frontend) : Au lieu d'un print() dans la console, le backend renvoie une réponse web structurée. On utilise la fonction jsonify() de Flask pour traduire un dictionnaire Python en texte universel JSON (JavaScript Object Notation), la langue comprise par tous les navigateurs internet.
   3. La Forteresse : Le backend s'exécute de manière invisible sur un serveur. Cela empêche les utilisateurs malveillants de voir vos mots de passe ou vos clés de connexion Azure SQL avec un simple clic droit sur le site.

## Structure d'un projet Backend professionnel (La modularité)
Dans l'industrie, on n'écrit pas tout dans un seul fichier. On fragmente le code en fichiers et en dossiers. Le fichier central s'appelle app.py. Il sert de chef d'orchestre et utilise l'importation (import) et les Blueprints de Flask pour assembler les fonctionnalités écrites dans d'autres fichiers (comme la gestion des produits ou l'authentification). Lorsque vous déployez, vous donnez l'intégralité de ce dossier au serveur, et le serveur sait qu'il doit exécuter app.py en premier.

## PARTIE 3 : CODES ET CONFIGURATIONS DE VOTRE PROJET HYBRIDE
Votre projet consiste à lier un formulaire web à un backend Flask Python, qui injecte les données dans Azure SQL (dans le Cloud de Microsoft), qui se synchronise ensuite automatiquement vers l'instance Microsoft SQL Server (DataGrid) sur votre ordinateur au Congo.
## 1. Le Code Backend Python (sqlconnection.py)
Ce code intègre la correction obligatoire pour intercepter un formulaire HTML classique sans utiliser JavaScript (request.form.get).

from flask import Flask, request, render_template_stringimport pyodbc
app = Flask(__name__)
# Chaîne de connexion fournie par Microsoft Azure# Remplacer les valeurs entre crochets par vos vrais identifiants AzureAZURE_CONN_STRING = (
    "Driver={ODBC Driver 18 for SQL Server};"
    "Server=tcp:votre-serveur-azure.database.windows.net,1433;"
    "Database=nom_de_votre_bdd;"
    "Uid=votre_utilisateur;"
    "Pwd=votre_mot_de_passe;"
    "Encrypt=yes;"
    "TrustServerCertificate=no;"
    "Connection Timeout=30;"
)

@app.route('/soumettre-formulaire', methods=['POST'])def soumettre_formulaire():
    # Capture des données du formulaire HTML classique (Pas de JavaScript)
    prenom = request.form.get('prenom')
    nom = request.form.get('nom')

    try:
        # 1. Connexion directe à la base de données SQL dans le Cloud Azure
        conn = pyodbc.connect(AZURE_CONN_STRING)
        cursor = conn.cursor()

        # 2. Requête SQL pour insérer les données
        query = "INSERT INTO Clients (Prenom, Nom) VALUES (?, ?)"
        cursor.execute(query, (prenom, nom))
        
        # 3. Validation et fermeture des tuyaux
        conn.commit()
        cursor.close()
        conn.close()

        return "<h1>Succès ! Vos données sont envoyées sur Azure SQL et se synchronisent en local !</h1>", 200

    except Exception as e:
        return f"<h1>Erreur de connexion Azure SQL : {str(e)}</h1>", 500
if __name__ == '__main__':
    app.run(debug=True)

## PARTIE 4 : CONFIGURATION ET MISE EN ROUTE DE NEW RELIC## Guide d'installation et de lancement
Pour activer la surveillance de New Relic sur votre projet local :

   1. Installez l'agent de surveillance dans votre terminal VS Code :
   
   pip install newrelic
   
   2. Récupérez votre clé de type INGEST - LICENSE depuis votre profil New Relic en ligne (Menu API Keys).
   3. Générez votre fichier de configuration local à la racine de votre projet :
   
   newrelic-admin generate-config VOTRE_CLÉ_DE_LICENCE_ICI newrelic.ini
   
   4. Ouvrez le fichier newrelic.ini et renommez votre application : app_name = Mon Formulaire Azure SQL.
   5. Coupez votre ancien serveur Flask dans le terminal en faisant Ctrl + C.
   6. Démarrez votre serveur à l'aide de la commande officielle de supervision :
   
   newrelic-admin run-program python sqlconnection.py
   
   7. Allez sur votre navigateur, remplissez et validez le formulaire 4 ou 5 fois. L'agent intercepte la vitesse de votre script Python et l'envoie automatiquement via des paquets web invisibles vers votre tableau de bord en ligne New Relic.

## PARTIE 6 : LA STRATÉGIE DE DÉPLOIEMENT EN PRODUCTION SUR AZURE
En développement (sur votre machine), vos fichiers partagent le même dossier. Mais une fois le site publié sur Internet (en production), ils se séparent pour habiter dans des services Cloud distincts d'Azure. Il n'est pas nécessaire de créer une Machine Virtuelle complexe.
## 1. La répartition des services sur Azure

* Le Visuel (HTML/CSS) $\rightarrow$ Azure Static Web Apps : Ce service stocke vos fichiers fixes de formulaire et les affiche instantanément aux visiteurs. Il est lié à votre dépôt GitHub Frontend.
* Les Calculs (Python) $\rightarrow$ Azure App Service (Option Python) : Un serveur géré (PaaS) par Microsoft qui maintient votre script Python allumé et actif 24h/24 sans que vous n'ayez de système d'exploitation Linux ou Windows Server à configurer. Il s'exécute de manière automatisée.
* La Base de données $\rightarrow$ Azure SQL : Votre base cloud actuelle.

## 2. Configuration DNS avec votre propre Nom de Domaine
Pour utiliser votre propre marque (ex: monprojet.com) à la place des adresses techniques d'Azure, vous devez vous connecter chez votre vendeur de domaine et ajouter deux enregistrements de type CNAME :

   1. Créez un CNAME avec le nom www pointant vers l'adresse technique d'Azure Static Web Apps. L'adresse de votre formulaire devient : https://monprojet.com.
   2. Créez un CNAME avec le nom api pointant vers l'adresse technique d'Azure App Service. L'adresse de votre backend devient : https://monprojet.com.

Votre ligne 360 en production deviendra alors définitivement stable, sécurisée (avec le cadenas HTTPS d'Azure gratuit) et performante :

<form action="https://monprojet.com" method="post" id="regForm" novalidate>
