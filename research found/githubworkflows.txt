Puisque vous maîtrisez déjà le concept sur Azure DevOps, ce guide va vous permettre de comprendre la philosophie de GitHub [github.com], sa syntaxe, et comment manipuler le terminal avec les commandes Git pour piloter vos automatisations comme un ingénieur DevOps.

##  COURS MAJEUR : MAÎTRISER LES WORKFLOWS ET GITHUB ACTIONS

## CHAPITRE 1 : PHILOSOPHIE ET ARCHITECTURE DE GITHUB ACTIONS## 1. Qu'est-ce que GitHub Actions ?
GitHub Actions est la plateforme de CI/CD native de GitHub [github.com]. Tout comme Azure Pipelines, elle permet d'automatiser vos phases de test, de compilation et de déploiement à chaque fois que vous modifiez votre code [github.com].
## 2. Le lexique universel de GitHub
Pour comprendre comment GitHub fonctionne en arrière-plan, il faut maîtriser ses 4 mots-clés :

* Workflow (Le flux complet) : C'est le processus automatisé global écrit dans un fichier au format YAML. Un projet peut avoir plusieurs workflows (un pour tester le code, un pour déployer sur Azure, un pour envoyer un e-mail).
* Event (Le déclencheur) : L'événement qui démarre le workflow (ex: un git push, une Pull Request, ou une heure précise de la journée).
* Runner (Le serveur éphémère) : C'est l'ordinateur distant (la machine virtuelle Linux ou Windows) que GitHub allume gratuitement pour vous afin d'y exécuter vos instructions [github.com].
* Job (Le groupe d'actions) : Une suite d'étapes exécutée sur un même Runner. Si vous avez deux Jobs indépendants, GitHub allumera deux serveurs en même temps pour diviser le temps de calcul.
* Step / Action (L'instruction précise) : Une commande de terminal ou un script exécuté dans le Job (ex: pip install).

## CHAPITRE 2 : LA STRUCTURE STRICTE DU REPERTOIRE
GitHub est extrêmement pointilleux sur l'emplacement de vos fichiers de configuration [github.com]. Si vous créez le fichier au mauvais endroit, il sera totalement invisible pour le serveur en ligne [github.com].
Le chemin d'accès doit obligatoirement être configuré ainsi à la racine de votre projet :

MonProjet/
│
├── .github/               <-- DOSSIER CACHÉ (Commence par un point)
│   └── workflows/         <-- SOUS-DOSSIER (Toujours au pluriel)
│       └── deploy.yml     <-- VOTRE FICHIER (Nom libre, extension .yml)
│
├── sqlconnection.py       <-- Votre code backend
└── formulaire.html        <-- Votre code frontend

## CHAPITRE 3 : EXPLICATION LIGNE PAR LIGNE DU FICHIER YAML
Voici le code standardisé pour votre application. Lisez attentivement l'explication technique de chaque bloc pour comprendre la "langue" de GitHub :

# 1. Le nom du script tel qu'il s'affichera dans l'onglet "Actions" de votre site webname: Pipeline Integration Continue Python
# 2. Le déclencheur (Event)on:
  push:
    branches: [ "main" ]  # Le script s'allume dès que vous validez un push sur main
# 3. Le bloc des serveurs à allumerjobs:
  verification_backend:
    runs-on: ubuntu-latest  # Allume une machine virtuelle Linux Ubuntu vierge
    
    # 4. La liste chronologique des actions à faire sur ce serveur
    steps:
      
      # Étape 1 : Le "Checkout". Une action officielle de GitHub qui permet à la 
      # machine virtuelle de cloner et de télécharger votre code actuel sur son disque.
      - name: Recupération du code du dépôt
        uses: actions/checkout@v4

      # Étape 2 : Configuration du moteur Python à l'intérieur du serveur Linux
      - name: Configuration de Python 3.x
        uses: actions/setup-python@v5
        with:
          python-version: '3.10'

      # Étape 3 : Installation de vos outils d'application (Flask, pyodbc, etc.)
      - name: Installation des dépendances du projet
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt
      # Étape 4 : L'inspection de votre fichier backend
      - name: Analyse syntaxique du code Python
        run: python -m py_compile sqlconnection.py

## CHAPITRE 4 : GUIDE PRATIQUE DU TERMINAL (LES COMMANDES)
Pour créer, configurer et envoyer ce workflow sur GitHub en utilisant uniquement votre ligne de commande, suivez scrupuleusement ces étapes dans votre terminal noir :
## 1. Créer la structure propre (sans fautes d'imbrication)
Nous allons utiliser mkdir -p pour créer les dossiers parents et enfants d'un seul coup :

mkdir -p .github/workflows

## 2. Créer le fichier YAML vide
Nous utilisons touch pour déposer notre feuille blanche au bon endroit :

touch .github/workflows/deploy.yml

## 3. Écrire le code
Ouvrez votre VS Code, cliquez sur le fichier deploy.yml situé dans l'arborescence .github/workflows/, collez-y le code du Chapitre 3 et sauvegardez avec Ctrl + S.
## 4. Le rituel de déploiement Git
Pour propulser ce nouveau workflow de votre ordinateur vers les serveurs de GitHub [github.com] :

# Vérifier que Git voit bien le nouveau dossier .github
git status
# Ajouter le fichier au panier de préparation
git add .github/workflows/deploy.yml
# Valider la modification avec un message explicite
git commit -m "DevOps: Ajout du pipeline de vérification syntaxique GitHub Actions"
# Propulser en ligne (Si vous utilisez GitHub pour ce dépôt)
git push origin main

## CHAPITRE 5 : VISUALISATION SUR L'INTERFACE GITHUB
Une fois le git push terminé, le spectacle commence en ligne :

   1. Connectez-vous sur votre compte GitHub [github.com] et ouvrez le dépôt (Repository) de votre projet.
   2. Regardez le menu horizontal tout en haut de la page (Code, Issues, Pull Requests...) et cliquez sur l'onglet Actions [github.com].
   3. Vous allez voir apparaître votre commit "DevOps: Ajout du pipeline..." avec un petit cercle orange clignotant. Cela signifie que le serveur de GitHub vient de s'allumer et exécute vos instructions en temps réel [github.com].
   4. Cliquez sur le titre de votre commit, puis dans le menu de gauche, cliquez sur verification_backend.
   5. L'écran noir de la console Linux s'ouvre. Vous voyez chaque ligne se dérouler, installer Flask, et analyser votre fichier sqlconnection.py.
   6. Si tout est correct, le cercle orange se transforme en un magnifique badge vert coché [github.com]. Votre code est officiellement certifié sans bug de syntaxe.

## Comparaison de fin d'étude : GitHub vs Azure DevOps
Pour ancrer définitivement vos connaissances, mémorisez ce tableau de correspondance des termes techniques :

| Action à accomplir | Syntaxe sur GitHub Actions | Syntaxe sur Azure Pipelines |
|---|---|---|
| Choisir l'OS du serveur | runs-on: ubuntu-latest | pool: vmImage: 'ubuntu-latest' |
| Lancer une commande brute | run: python script.py | script: python script.py |
| Utiliser un script préconfiguré | uses: actions/checkout@v4 | - task: UsePythonVersion@0 |
